## 1. 语义对象模型

### 1.1 用户 (User)

**业务含义**: 系统的参与者，包括普通用户（学生/教职工）和管理员。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 用户唯一标识（自增主键） |
| openid | TEXT | 微信OpenID，用于身份认证和消息推送 |
| student_no | TEXT | 学号（学生用户） |
| staff_no | TEXT | 工号（教职工用户） |
| real_name | TEXT | 真实姓名，用于身份核验 |
| mobile | TEXT | 手机号码，联系方式 |
| nickname | TEXT | 昵称，显示名称 |
| avatar_url | TEXT | 头像URL地址 |
| college | TEXT | 所属学院 |
| class_name | TEXT | 班级名称 |
| role | INTEGER | 角色：1-普通用户，2-管理员 |
| status | INTEGER | 状态：1-正常，2-禁用 |
| created_at | TEXT | 创建时间（ISO 8601格式） |
| updated_at | TEXT | 最后更新时间（ISO 8601格式） |

**业务规则**:
- 每个用户必须有唯一的微信OpenID
- 管理员账号由系统初始化时自动创建
- 被禁用的用户无法发布新内容

---

### 1.2 物品分类 (ItemCategory)

**业务含义**: 对失物/招领物品进行分类管理，便于检索和统计。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 分类唯一标识（自增主键） |
| name | TEXT | 分类名称（如：证件、电子产品、衣物等） |
| status | INTEGER | 状态：1-启用，0-禁用 |
| sort_no | INTEGER | 排序号，数值越小越靠前 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 最后更新时间 |

**业务规则**:
- 分类名称必须唯一
- 系统初始化时预置6个默认分类：证件、电子产品、衣物、书籍、钥匙、数码配件
- 禁用的分类不再出现在发布表单中

---

### 1.3 物品帖子 (ItemPost)

**业务含义**: 用户发布的失物或招领信息，是系统的核心业务对象。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 帖子唯一标识（自增主键） |
| post_no | TEXT | 帖子编号，业务唯一标识（格式：前缀+时间戳+UUID片段） |
| post_type | INTEGER | 帖子类型：1-失物（我丢了），2-招领（我捡到了） |
| title | TEXT | 标题，简要描述物品 |
| category_id | INTEGER | 所属分类ID（外键） |
| occur_time | TEXT | 发生时间（丢失时间或拾取时间） |
| occur_location | TEXT | 发生地点（丢失地点或拾取地点） |
| detail | TEXT | 详细描述，包括物品特征等 |
| contact_info | TEXT | 联系方式 |
| reward_amount | REAL | 悬赏金额（仅失物帖有效） |
| deposit_location | TEXT | 存放地点（招领帖使用，表示物品存放在哪里） |
| publisher_id | INTEGER | 发布者ID（外键关联users表） |
| status | INTEGER | 状态：0-待审核，1-审核中，2-已发布，3-已驳回，4-已认领/已找到，7-已删除 |
| reject_reason | TEXT | 驳回原因（status=3或7时填写） |
| audit_admin_id | INTEGER | 审核管理员ID（外键） |
| audit_at | TEXT | 审核时间 |
| published_at | TEXT | 发布时间（审核通过的时间） |
| expire_at | TEXT | 过期时间（发布后90天） |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 最后更新时间 |

**业务规则**:
- 帖子编号全局唯一，格式为：`POST{timestamp}{uuid片段}`
- 新发布的帖子状态为0（待审核），需要管理员审核通过后才能公开显示
- 审核超时时间为24小时
- 帖子有效期为90天，过期后不再显示
- 只有status=2的帖子可以被认领
- 认领成功后帖子状态更新为4

---

### 1.4 帖子图片 (ItemPostImage)

**业务含义**: 与帖子关联的图片证据，支持多图上传。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 图片记录唯一标识（自增主键） |
| post_id | INTEGER | 关联的帖子ID（外键） |
| img_url | TEXT | 图片URL地址 |
| sort_no | INTEGER | 排序号，控制图片显示顺序 |
| created_at | TEXT | 上传时间 |

**业务规则**:
- 每个帖子最多可上传5张图片
- 第一张图片作为封面图在列表中显示
- 删除帖子时级联删除相关图片

---

### 1.5 认领申请 (ClaimApplication)

**业务含义**: 用户对招领帖子发起的认领申请，记录认领流程和审核状态。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 申请唯一标识（自增主键） |
| apply_no | TEXT | 申请编号，业务唯一标识（格式：C+时间戳+UUID片段） |
| post_id | INTEGER | 关联的帖子ID（外键） |
| applicant_id | INTEGER | 申请人ID（外键） |
| verify_desc | TEXT | 验证描述，申请人提供的物品特征证明 |
| evidence_images | TEXT | 证据图片JSON数组字符串 |
| status | INTEGER | 状态：1-待审核，2-已通过，3-已拒绝/已撤销 |
| reviewer_id | INTEGER | 审核人ID（外键，通常是帖子发布者或管理员） |
| review_remark | TEXT | 审核备注/拒绝原因 |
| reviewed_at | TEXT | 审核时间 |
| expired_at | TEXT | 过期时间（提交后72小时） |
| created_at | TEXT | 申请时间 |
| updated_at | TEXT | 最后更新时间 |

**业务规则**:
- 申请编号全局唯一，格式为：`C{timestamp}{uuid片段}`
- 只能对status=2（已发布）的帖子发起认领申请
- 认领申请处理超时时间为72小时
- 申请人可以撤销status=1的申请
- 认领通过后，关联帖子状态更新为4（已认领）
- evidence_images存储为JSON字符串格式

---

### 1.6 审核日志 (PostAuditLog)

**业务含义**: 记录帖子审核的历史操作，用于追溯和审计。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 日志唯一标识（自增主键） |
| post_id | INTEGER | 关联的帖子ID（外键） |
| admin_id | INTEGER | 操作管理员ID（外键） |
| action | INTEGER | 操作类型：1-通过，2-驳回，3-删除 |
| reason | TEXT | 操作原因/备注 |
| created_at | TEXT | 操作时间 |

**业务规则**:
- 每次审核操作都会生成一条日志
- 支持追溯帖子的完整审核历史

---

### 1.7 公告 (Announcement)

**业务含义**: 系统发布的公告通知，向所有用户展示重要信息。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 公告唯一标识（自增主键） |
| title | TEXT | 公告标题 |
| content | TEXT | 公告内容（富文本） |
| status | INTEGER | 状态：1-已发布，0-草稿 |
| publish_at | TEXT | 发布时间 |
| created_by | INTEGER | 创建者ID（管理员） |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 最后更新时间 |

**业务规则**:
- 只有管理员可以发布公告
- 首页显示最新的有效公告

---

### 1.8 通知 (Notification)

**业务含义**: 向用户推送的系统通知，包括审核结果、认领状态变更等。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 通知唯一标识（自增主键） |
| user_id | INTEGER | 接收用户ID（外键） |
| biz_type | INTEGER | 业务类型：1-审核通知，2-公告通知，3-认领通知 |
| biz_id | INTEGER | 关联的业务ID（帖子ID或申请ID） |
| title | TEXT | 通知标题 |
| content | TEXT | 通知内容 |
| is_read | INTEGER | 是否已读：0-未读，1-已读 |
| read_at | TEXT | 阅读时间 |
| created_at | TEXT | 创建时间 |

**业务规则**:
- 通知触发场景：
  - 帖子审核通过/驳回
  - 收到新的认领申请
  - 认领申请状态变更
- 支持站内通知、微信订阅消息、企业微信应用消息三种推送方式
- 用户可以标记通知为已读

---

### 1.9 反馈 (Feedback)

**业务含义**: 用户向系统提交的意见建议或问题反馈。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 反馈唯一标识（自增主键） |
| user_id | INTEGER | 提交用户ID（外键） |
| content | TEXT | 反馈内容 |
| status | INTEGER | 状态：1-待处理，2-已回复 |
| reply_content | TEXT | 回复内容 |
| replied_by | INTEGER | 回复者ID（管理员） |
| replied_at | TEXT | 回复时间 |
| created_at | TEXT | 提交时间 |

**业务规则**:
- 用户可以提交反馈但不一定能看到回复状态
- 管理员可以回复反馈

---

### 1.10 系统配置 (SystemConfig)

**业务含义**: 系统运行时可配置的参数，无需修改代码即可调整业务规则。

| 属性 | 数据类型 | 业务说明 |
|------|---------|---------|
| id | INTEGER | 配置唯一标识（自增主键） |
| config_key | TEXT | 配置键名（唯一） |
| config_value | TEXT | 配置值 |
| description | TEXT | 配置说明 |
| updated_at | TEXT | 最后更新时间 |

**预设配置项**:

| 配置键 | 配置值 | 说明 |
|-------|-------|------|
| max_daily_publish | 10 | 每日最大发布数 |
| audit_timeout_hours | 24 | 审核超时小时数 |
| claim_timeout_hours | 72 | 认领处理超时小时数 |
| post_expire_days | 90 | 信息过期天数 |

**业务规则**:
- 配置键名必须唯一
- 只有管理员可以修改配置
- 配置修改后立即生效

### 2.2 关系详细说明

#### 2.2.1 用户与帖子 (users ↔ item_posts)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个用户可以发布多个帖子（失物或招领信息）
- 每个帖子必须由一个用户发布
- 外键: `item_posts.publisher_id → users.id`

**基数约束**: 
- 最小基数: 0..N（用户可以不发布任何帖子）
- 最大基数: 1..N（帖子必须有发布者）

---

#### 2.2.2 分类与帖子 (item_categories ↔ item_posts)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个分类可以包含多个帖子
- 每个帖子属于一个分类（可选）
- 外键: `item_posts.category_id → item_categories.id`

**基数约束**:
- 最小基数: 0..N（分类可以没有帖子）
- 最大基数: 0..1（帖子可以不选择分类）

---

#### 2.2.3 帖子与图片 (item_posts ↔ item_post_images)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个帖子可以关联多张图片（最多5张）
- 每张图片只属于一个帖子
- 外键: `item_post_images.post_id → item_posts.id`

**基数约束**:
- 最小基数: 0..N（帖子可以没有图片）
- 最大基数: 1..5（图片必须在1-5张之间）

---

#### 2.2.4 用户与认领申请 (users ↔ claim_applications)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个用户可以提交多个认领申请
- 每个申请由一个用户提交
- 外键: `claim_applications.applicant_id → users.id`

**基数约束**:
- 最小基数: 0..N（用户可以不提交申请）
- 最大基数: 1..N（申请必须有申请人）

---

#### 2.2.5 帖子与认领申请 (item_posts ↔ claim_applications)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个帖子可以收到多个认领申请
- 每个申请针对一个帖子
- 外键: `claim_applications.post_id → item_posts.id`

**基数约束**:
- 最小基数: 0..N（帖子可以没有申请）
- 最大基数: 1..N（申请必须关联帖子）

**业务约束**: 只有status=2（已发布）的帖子才能被认领

---

#### 2.2.6 管理员与审核日志 (users ↔ post_audit_logs)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个管理员可以执行多次审核操作
- 每条审核日志由一个管理员创建
- 外键: `post_audit_logs.admin_id → users.id`

**基数约束**:
- 最小基数: 0..N（管理员可以没有审核记录）
- 最大基数: 1..N（日志必须有操作者）

---

#### 2.2.7 帖子与审核日志 (item_posts ↔ post_audit_logs)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个帖子可以有多条审核日志（多次审核历史）
- 每条日志关联一个帖子
- 外键: `post_audit_logs.post_id → item_posts.id`

**基数约束**:
- 最小基数: 0..N（帖子可能未被审核）
- 最大基数: 1..N（日志必须关联帖子）

---

#### 2.2.8 用户与通知 (users ↔ notifications)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个用户可以接收多条通知
- 每条通知发送给一个用户
- 外键: `notifications.user_id → users.id`

**基数约束**:
- 最小基数: 0..N（用户可以没有通知）
- 最大基数: 1..N（通知必须有接收者）

---

#### 2.2.9 管理员与公告 (users ↔ announcements)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个管理员可以发布多个公告
- 每个公告由一个管理员创建
- 外键: `announcements.created_by → users.id`

**基数约束**:
- 最小基数: 0..N（管理员可以不发布公告）
- 最大基数: 0..1（公告可以有创建者，但允许为空）

---

#### 2.2.10 用户与反馈 (users ↔ feedbacks)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个用户可以提交多个反馈
- 每个反馈由一个用户提交
- 外键: `feedbacks.user_id → users.id`

**基数约束**:
- 最小基数: 0..N（用户可以不提交反馈）
- 最大基数: 1..N（反馈必须有提交者）

---

#### 2.2.11 管理员与反馈回复 (users ↔ feedbacks.reply)

**关系类型**: 1:N (一对多)

**关系描述**:
- 一个管理员可以回复多个反馈
- 每个反馈回复由一个管理员执行
- 外键: `feedbacks.replied_by → users.id`

**基数约束**:
- 最小基数: 0..N（管理员可以不回复反馈）
- 最大基数: 0..1（反馈可以未回复）

```

```

### 3.2 详细关系模式

#### 3.2.1 users (用户表)

```sql
CREATE TABLE users (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  openid       TEXT    UNIQUE NOT NULL,
  student_no   TEXT,
  staff_no     TEXT,
  real_name    TEXT,
  mobile       TEXT,
  nickname     TEXT,
  avatar_url   TEXT,
  college      TEXT,
  class_name   TEXT,
  role         INTEGER DEFAULT 1,      -- 1: 普通用户, 2: 管理员
  status       INTEGER DEFAULT 1,      -- 1: 正常, 2: 禁用
```

## 4. 索引设计

### 4.1 已创建的索引

系统在数据库中创建了以下索引以优化查询性能：

#### 4.1.1 idx_posts_type_status

```sql
CREATE INDEX idx_posts_type_status ON item_posts(post_type, status, created_at);
```

**用途**: 优化按帖子类型和状态筛选的查询，并按创建时间排序  
**适用场景**:
- 首页加载：获取所有已发布的失物/招领帖子
- 分类浏览：按类型筛选并显示最新帖子
- 典型查询: `SELECT * FROM item_posts WHERE post_type=? AND status=? ORDER BY created_at DESC`

**覆盖列**: `(post_type, status, created_at)` - 复合索引，最左前缀原则

---

#### 4.1.2 idx_posts_publisher

```sql
CREATE INDEX idx_posts_publisher ON item_posts(publisher_id, status);
```

**用途**: 优化查询某个用户发布的所有帖子  
**适用场景**:
- "我的发布"页面：获取当前用户发布的所有帖子
- 用户个人中心：统计用户发布的帖子数量
- 典型查询: `SELECT * FROM item_posts WHERE publisher_id=? AND status=?`

**覆盖列**: `(publisher_id, status)` - 复合索引

---

#### 4.1.3 idx_claims_post_status

```sql
CREATE INDEX idx_claims_post_status ON claim_applications(post_id, status);
```

**用途**: 优化查询某个帖子的认领申请状态  
**适用场景**:
- 帖子详情页：检查是否有待处理的认领申请
- 管理员审核：获取某帖子的所有待审核申请
- 典型查询: `SELECT * FROM claim_applications WHERE post_id=? AND status=?`

**覆盖列**: `(post_id, status)` - 复合索引

---

#### 4.1.4 idx_notifications_user

```sql
CREATE INDEX idx_notifications_user ON notifications(user_id, is_read, created_at);
```

**用途**: 优化用户通知列表查询和未读计数  
**适用场景**:
- 消息中心：获取用户的所有通知并按时间排序
- 未读计数：统计用户的未读通知数量
- 典型查询: 
  - `SELECT * FROM notifications WHERE user_id=? ORDER BY created_at DESC`
  - `SELECT COUNT(*) FROM notifications WHERE user_id=? AND is_read=0`

**覆盖列**: `(user_id, is_read, created_at)` - 复合索引

---

### 4.2 隐式索引

SQLite会自动为以下列创建索引：

1. **主键索引**: 所有表的 `id` 列（INTEGER PRIMARY KEY）
2. **唯一约束索引**:
   - `users.openid`
   - `item_posts.post_no`
   - `item_categories.name`
   - `claim_applications.apply_no`
   - `system_configs.config_key`

---

### 4.3 索引使用建议

**已有索引的优点**:
- 覆盖了主要的查询场景
- 复合索引遵循最左前缀原则
- 避免了全表扫描

**潜在优化空间** (未来可以考虑添加):
- `item_posts.status` 单独索引（如果需要频繁按状态过滤）
- `claim_applications.applicant_id` 索引（如果"我的申请"查询频繁）
- `notifications.is_read` 单独索引（如果未读计数查询频繁）

### 5.2 业务规则汇总

#### 5.2.1 发布规则

1. **每日发布限制**: 每个用户每天最多发布 `max_daily_publish` (默认10) 条帖子
2. **审核机制**: 所有新发布的帖子必须经过管理员审核才能公开显示
3. **审核超时**: 帖子应在 `audit_timeout_hours` (默认24) 小时内完成审核
4. **有效期**: 帖子发布后 `post_expire_days` (默认90) 天后过期
5. **图片限制**: 每个帖子最多上传5张图片

---

#### 5.2.2 认领规则

1. **认领条件**: 只能对 `status=2` (已发布) 的帖子发起认领申请
2. **处理超时**: 认领申请应在 `claim_timeout_hours` (默认72) 小时内处理
3. **撤销权限**: 申请人可以撤销 `status=1` (待审核) 的申请
4. **认领成功后**: 帖子状态从2更新为4 (已认领)

---

#### 5.2.3 通知规则

1. **自动触发**: 以下情况自动发送通知
   - 帖子审核通过/驳回 → 通知发布者
   - 收到新的认领申请 → 通知帖子发布者
   - 认领申请状态变更 → 通知申请人
2. **推送渠道**: 
   - 站内通知（必发）
   - 微信订阅消息（需配置且用户授权）
   - 企业微信应用消息（需配置且有用户账号）

---

#### 5.2.4 权限规则

1. **普通用户 (role=1)**:
   - 可以发布、编辑、删除自己的帖子
   - 可以发起认领申请
   - 可以查看公开的帖子和公告
   - 可以提交反馈

2. **管理员 (role=2)**:
   - 拥有普通用户的所有权限
   - 可以审核帖子（通过/驳回）
   - 可以处理认领申请
   - 可以发布公告
   - 可以管理用户（禁用/解禁）
   - 可以查看统计数据
   - 可以修改系统配置
   - 可以回复反馈

## 附录

### A. 数据库版本信息

- **数据库引擎**: SQLite 3.x
- **Node.js驱动**: better-sqlite3
- **Journal模式**: WAL (Write-Ahead Logging)
- **字符编码**: UTF-8

### B. 命名规范

- **表名**: 小写字母 + 下划线分隔（snake_case），复数形式
- **字段名**: 小写字母 + 下划线分隔（snake_case）
- **主键**: 统一使用 `id`
- **外键**: `{关联表名}_id` 格式
- **时间字段**: `{事件}_at` 格式，存储ISO 8601格式字符串

### C. 时间格式

所有时间字段均使用 **ISO 8601** 格式：
```
YYYY-MM-DDTHH:mm:ss.sssZ
示例: 2026-04-09T12:30:45.123Z
```

### D. 编号生成规则

系统使用 `genNo(prefix)` 函数生成业务编号：

```javascript
// 格式: {prefix}{timestamp}{uuid片段(6位)}
post_no = "POST" + Date.now() + UUID.slice(0, 6).toUpperCase()
apply_no = "C" + Date.now() + UUID.slice(0, 6).toUpperCase()
```

示例:
- 帖子编号: `POST1712649045123A1B2C3`
- 申请编号: `C1712649045123D4E5F6`

### E. 数据库维护建议

2. **清理过期数据**: 定期清理 `expire_at < 当前时间` 的帖子
3. **归档历史数据**: 将一年前的数据归档到历史库
4. **监控数据库大小**: 当数据库超过500MB时考虑清理或归档
5. **VACUUM命令**: 定期执行 `VACUUM` 回收空间
